01 — System Overview

Three Operational Workflows

Every run of this pipeline falls into one of three modes. Training builds the models, Inference applies them weekly, and Backtesting simulates historical performance. They share the same data layer and feature engineering code but diverge at the scoring step.

🏋️

Training

Builds and serializes all model artifacts. Runs once on historical data; subsequent runs can load from cache using --skip_fetch.

pipeline/train.py
📡

Inference

Runs weekly. Fetches a fresh data snapshot — never re-uses the training panel tail — to avoid stale or leakage-prone data.

pipeline/infer.py
📊

Backtesting

Simulates portfolio changes on historical data with real execution cost modeling. Outputs HTML performance reports and equity curve parquet.

03 — Configuration

MarketConfig — Single Source of Truth

MarketConfig is a dataclass defined in pipeline/config/base.py. Every downstream module reads from it rather than maintaining its own constants, so swapping from NSE to S&P 500 requires zero changes to model or backtest code.

04 — Data Layer

Panel Construction Pipeline

The master panel is a MultiIndex(date, ticker) DataFrame. Every piece of downstream logic — features, targets, model scoring — operates on this structure. Getting this right is the foundation of the entire system.

Universe Eligibility

UniverseBuilder marks rows tradable monthly (first trading day) using: min 20-day ADV, min market cap, min trading history, and delisting check. Without a symbol master, sector defaults to Unknown and market cap may be NaN, blocking all stocks.

Data Source Priority

Primary and fallback sources are set per MarketConfig. For S&P 500 and NASDAQ: Polygon → Tiingo. FX retrieval falls back to a static 1/75 constant if live rates fail (NSE only).

05 — Feature Engineering

Three-Layer Feature System

Features are built in three separate passes and combined into the panel. All features_* columns are winsorized cross-sectionally per date at the 1st and 99th percentile to prevent outlier domination.

06 — Model Layer

Ensemble Architecture

Three models are combined into a single blended score. LightGBM drives the ranking, CatBoost adds probability calibration, and inverse-volatility provides a risk-aware tilt.

Purged Walk-Forward Cross-Validation

The CV scheme prevents future leakage by inserting a purge window (40 days) and embargo window (5 days) between training and test folds.

07 — Portfolio Construction

From Score to Weights

PortfolioConstructor applies a cascade of filters and constraints to convert a raw score vector into investable weights.

09 — Monitoring & Retraining

Feature Drift Detection (PSI)

FeatureDriftMonitor compares the live feature distribution against the training baseline using Population Stability Index. Three thresholds trigger different actions.

If more than 20% of monitored features breach the retrain threshold simultaneously, the monitor logs a global retrain trigger in addition to the per-feature flags.

10 — Implementation Caveats

Known Issues & Gotchas

These are the most important pitfalls to be aware of before running or extending the pipeline.

🔑
No tickers file = demo mode only Without a --tickers_file, the pipeline falls back to the benchmark ticker alone. That is not a realistic equity-selection universe.
⚠️
missing features_ prefix on multi-timeframe columns weekly_trend and monthly_trend lack the features_ prefix. They are silently excluded from model training unless renamed or explicitly included.
🚨
Potential leakage: future_vol_20d in inference The ensemble uses future_vol_20d when available. This is a forward-looking target. If the column is present during live inference it introduces look-ahead bias. The code neutralizes it when absent, but the column should never be passed at inference time.
📐
Overlapping return periods in backtest The backtest uses future_20d_return (20-day horizon) while rebalancing weekly. Consecutive weekly groups have overlapping 20-day windows; treat the interpretation of annualized metrics carefully.
🏗️
No market cap without shares_outstanding PanelConstructor.build only computes market_cap_usd when shares_outstanding is provided. Without it, UniverseBuilder may exclude every ticker, producing an empty universe.
📉
Calibration fit on full training data The final calibrator is fit and validated on the same full-dataset predictions rather than a held-out calibration fold. This may overstate calibration quality.
🌐
Chart.js CDN dependency The HTML performance report loads Chart.js from a CDN. Charts will not render when the report is opened offline.
🔤
Mojibake in source comments Several source files have encoding issues causing garbled characters in comments. The code executes correctly but cleaning file encodings would improve maintainability.

Adding a New Market — 6 Steps

1

Create pipeline/config/{market}.py

Instantiate MarketConfig with the correct calendar, benchmark ticker, currency, data sources, slippage, liquidity, and risk settings.

2

Register in __init__.py

Import the new config and add it to MARKET_CONFIGS so get_config() can resolve it.

3

Verify data provider compatibility

Ensure the chosen primary/fallback source can fetch tickers in the format used by that market (e.g. NSE suffix conventions).

4

Provide a ticker file and symbol master

Supply a flat ticker list and, ideally, a symbol master CSV with sector, listing date, and shares outstanding for proper universe eligibility and market cap calculation.

5

Run test_integration.py

Use the synthetic-data smoke test to validate that all major pipeline components initialize and run end-to-end without live data.

6

Run training with --skip_fetch=False

Execute a full training run to build and save artifacts. All downstream modules read from MarketConfig and require no further changes unless the data source format differs.
